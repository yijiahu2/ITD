

**当前项目最大风险不是某个模块没做，而是阶段性落地后留下了多条入口、多套流程、多处重复职责。如果不收口，后面会变成“能跑但不可控”的智能体项目。**

# 一、必须先定最终总入口：阶段入口全部降级或删除

现在至少有两条主线：

1. 旧主线：`ITD_agent/orchestration/orchestrator.py`
   这是大总管，承担输入、处理、规划、主模型、专家模型、评估、局部修正、finetune 推荐、输出汇总等大量职责。
2. 新阶段主线：`ITD_agent/evolution/cli.py`
   里面把 `evolve-infer`、`review`、`finetune-pool`、`train` 都挂到一个 CLI 上。

这两个入口不能长期并存。否则会出现：

| 问题                                                         | 后果                                     |
| ------------------------------------------------------------ | ---------------------------------------- |
| `orchestration` 能跑完整流程，`evolution/cli.py` 也能跑分阶段流程 | 用户和 Codex 不知道该改哪条主线          |
| V1/V2/V3 阶段性脚本继续保留                                  | 功能重复、测试重复、配置重复             |
| 同一个功能既在 orchestration 又在 evolution/training_loop 中存在 | 后续改一处，另一处失效                   |
| CLI 挂在 `evolution/cli.py` 下                               | 从命名上看像 V1 模块入口，不像全项目入口 |
| `orchestrator.py` 太重                                       | 最终会变成智能体项目里的“上帝文件”       |
| 旧阶段产物没有归档/删除                                      | 项目越来越像补丁堆叠，不像闭环系统       |

## 解决方案

最终只能保留一个正式入口：

```text
ITD_agent/cli/
  __init__.py
  main.py
  commands/
    run.py
    review.py
    train.py
    state.py
    export.py
```

最终命令面应该是：

```bash
itd-agent run --config xxx.yaml
itd-agent review --config xxx.yaml
itd-agent train --config xxx.yaml
itd-agent state --db xxx.sqlite
itd-agent export --run-dir xxx --out xxx
```

而不是继续暴露：

```bash
itd-agent evolve-infer
itd-agent review run
itd-agent finetune-pool export
itd-agent train run
```

原因很简单：
`evolve-infer` 是阶段名，不是最终产品名。最终用户不应该关心 V1/V2/V3，只应该关心“运行、审查、训练、导出、查询状态”。

# 二、`evolution/`、`orchestration/`、`training_loop/`、`finetune_pool/`、`cli/` 最终边界

我建议最终边界这样定。

## 1. `cli/`：只做入口，不做业务

### 最终职责

`cli/` 只负责：

| 职责                  | 说明                                    |
| --------------------- | --------------------------------------- |
| 解析命令行参数        | `--config`、`--run-id`、`--db`、`--out` |
| 调用 orchestration 层 | 不直接调用 segmentation/training/fusion |
| 打印 JSON 或摘要      | 只负责输出给用户                        |
| 退出码管理            | 成功/失败                               |

### 禁止事项

`cli/` 不应该：

- 不做 COCO 解析；
- 不做 ROI；
- 不做专家模型路由；
- 不做训练判断；
- 不直接写 finetune pool；
- 不直接跑模型。

### 需要处理的现有文件

当前 `ITD_agent/evolution/cli.py` 应该迁移为：

```text
ITD_agent/cli/main.py
```

迁移后，**原 `ITD_agent/evolution/cli.py` 不建议保留 shim**。因为你已经明确说不要无限套娃，所以这个文件应该删除，而不是留下：

```python
from ITD_agent.cli.main import main
```

这种薄 shim 只会继续制造混乱。

## 2. `orchestration/`：最终闭环编排层，不做具体算法

### 最终职责

`orchestration/` 应该是整个智能体闭环的总控制器，负责：

```text
输入规范化
→ 数据处理
→ 主模型推理
→ 评估诊断
→ ROI/专家是否介入
→ 专家模型调用
→ 融合/回滚
→ 轨迹记录
→ V2 review
→ V3 training trigger
→ 输出发布
```

但是注意：
它只负责**调用顺序**，不负责每个环节的具体计算。

### 应保留的文件方向

```text
ITD_agent/orchestration/
  workflow.py          # 最终主流程 run_full_workflow()
  run_context.py       # RunContext、路径、run_id、状态
  stage_runner.py      # 各阶段调用封装
  recovery.py          # 失败恢复/断点
```

### 应删除或拆解的现有问题

`orchestration/orchestrator.py` 现在太重。它不能继续作为最终上帝文件。

应该拆成：

| 当前职责                          | 应迁移到                                 |
| --------------------------------- | ---------------------------------------- |
| 输入 manifest / runtime normalize | `input_layer`                            |
| 语义先验 / segmentation 执行      | `segmentation`                           |
| 主模型评估                        | `evaluation_analysis`                    |
| ROI 评估                          | `evaluation_analysis` 或 `evolution/roi` |
| 专家模型计划                      | `planning`                               |
| 专家模型执行                      | `evolution/expert`                       |
| 融合/回滚                         | `evolution/fusion`                       |
| finetune 推荐                     | `finetune_pool`                          |
| 训练触发                          | `training_loop`                          |
| 输出发布                          | `output_layer`                           |

`orchestration` 最终只保留类似：

```python
def run_workflow(config_path):
    ctx = build_run_context(config_path)
    input_result = input_layer.prepare(ctx)
    processing_result = data_processing.prepare(ctx, input_result)
    main_result = segmentation.run_main(ctx)
    eval_result = evaluation_analysis.evaluate_main(ctx, main_result)
    evolution_result = evolution.adapt_if_needed(ctx, eval_result)
    review_result = finetune_pool.review_and_store(ctx, evolution_result)
    training_result = training_loop.maybe_train(ctx, review_result)
    output_result = output_layer.publish(ctx)
    return final_summary
```

这是最容易混乱的模块。

### 最终职责

`evolution/` 应该负责：

| 子能力                      | 是否属于 evolution       |
| --------------------------- | ------------------------ |
| 主模型错误分解后的 ROI 候选 | 属于                     |
| ROI 聚类                    | 属于                     |
| 专家模型任务构建            | 属于                     |
| 专家模型执行记录            | 属于                     |
| 专家结果和主模型结果比较    | 属于                     |
| 局部融合 / 回滚             | 属于                     |
| trajectory 写入             | 属于                     |
| SQLite 状态记录             | 属于                     |
| 训练样本候选标记            | 可以产生候选，但不能训练 |
| 模型训练                    | 不属于                   |
| 模型注册                    | 不属于                   |
| 主模型/专家模型权重更新     | 不属于                   |
| CLI 入口                    | 不属于                   |

当前 `evolve_infer_runner.py` 把 V1 主流程写得比较完整：加载 COCO、运行主模型、错误分解、构建 ROI、专家任务、专家执行、比较、融合、写 trajectory、写 state。

但从最终架构看，`run_evolve_infer_v1` 这个名字应该废弃。最终应该改成：

```text
evolution/adaptive_inference.py
  run_adaptive_inference_stage(...)
```

不要保留 `v1` 命名。
因为最终项目不应该让核心函数永远叫 `v1`。

| 当前文件/函数                | 最终处理                                               |
| ---------------------------- | ------------------------------------------------------ |
| `evolution/cli.py`           | 移到 `cli/main.py` 后删除原文件                        |
| `run_evolve_infer_v1`        | 改为 `run_adaptive_inference_stage`                    |
| `config_preflight_v1` 类命名 | 改为 `preflight_runtime_config`                        |
| `review/` 如果是 V2 审查     | 应迁到 `finetune_pool/review` 或 `memory_store/review` |
| `training_loop` 相关 dry-run | 只保留 candidate 标记，不做 trigger 决策               |

## 4. `finetune_pool/`：样本池与审查层，不负责训练

### 最终职责

`finetune_pool/` 应该负责：

| 职责                            | 说明                       |
| ------------------------------- | -------------------------- |
| 接收 evolution 产生的候选样本   | ROI、专家修正、失败样本    |
| 样本质量筛选                    | 可训练/不可训练/需人工审查 |
| 样本聚类                        | 避免重复样本               |
| 样本索引                        | 可检索、可追踪             |
| 数据集导出                      | 给 training_loop 使用      |
| 记录 rejected/accepted/deferred | 保证可审计                 |

### 禁止事项

`finetune_pool/` 不应该：

- 不跑训练；
- 不决定模型晋升；
- 不改 route map；
- 不直接调用主模型/专家模型；
- 不做 runtime orchestration。

你现在的 `finetune_pool` 已经有 `dataset_exporter.py`、`policy.py`、`recommendation.py`、`store.py` 等文件，方向是对的。([github.com](https://github.com/yijiahu2/ITD/tree/main/ITD_agent/finetune_pool))

但建议把 V2 review 相关能力从 `evolution/review` 收口进来。
因为从最终闭环看，review 的本质不是 evolution，而是**样本入池前的审查与沉淀**。

推荐结构：

```text
ITD_agent/finetune_pool/
  intake.py
  review/
    reviewer.py
    rules.py
    human_review.py
  store.py
  policy.py
  clusterer.py
  dataset_exporter.py
  recommendation.py
```

这样 V2 不再是一个阶段模块，而是 finetune pool 的正式组成部分。

## 5. `training_loop/`：只负责受控训练与模型候选，不负责推理期决策

### 最终职责

`training_loop/` 应该负责：

| 职责                          | 说明                              |
| ----------------------------- | --------------------------------- |
| 从 finetune_pool 读取训练样本 | 不直接从 evolution 乱拿           |
| 样本质量门控                  | `quality_gate`                    |
| replay guard                  | 防止灾难性退化                    |
| pilot training                | 小规模试训                        |
| formal training               | 正式训练                          |
| post-train evaluation         | 训练后评估                        |
| model registry                | 注册候选模型                      |
| promotion suggestion          | 给出晋升建议                      |
| routing update candidate      | 只生成候选，不直接改 active route |

当前 `training_runner.py` 已经有这些东西：V2 assets 加载、finetune bundle、质量门控、trigger、pilot/formal training、replay guard、DOM geometry guard、模型注册、晋升判断、routing candidate、feedback。

所以 `training_loop` 的方向是正确的。

但需要改一个关键点：

**training_loop 不应该直接理解 “V2 review dir” 这个阶段概念。**

最终应该改成：

```yaml
source:
  finetune_pool_dir: ...
  sample_query: ...
  replay_pool_dir: ...
```

而不是：

```yaml
source:
  v2_review_dir: ...
```

原因：V2 是阶段名，不是最终产品模块名。

# 三、`llm_gateway/` 现在确实像“没参与主闭环”

你这个感觉是对的。

我查到 `input_layer/adapters.py` 里确实有 `llm_gateway` 配置解析，比如 provider、model、base_url 这些字段会被塞进 runtime config。

但从目前已审查到的主流程看，`llm_gateway` 没有成为一个清晰的独立调用层。它更像是：

```text
配置里有 llm_gateway 字段
planning/orchestration 可能会用
但模块职责没有真正立起来
```

这会导致一个严重问题：

**大模型在项目中到底是“决策顾问层”，还是只是某些 planner 里的可选开关？现在边界不够清楚。**

## 最终 `llm_gateway/` 应该承担什么？

它应该只做四件事：

| 职责                | 说明                                               |
| ------------------- | -------------------------------------------------- |
| 统一 LLM 调用       | Doubao/OpenAI/本地模型统一接口                     |
| Prompt 输入输出协议 | 输入 context，输出结构化 JSON                      |
| 失败降级            | LLM 不可用时回退规则引擎                           |
| 审计记录            | 记录 prompt、response、token、latency、decision_id |

## 它不应该做什么？

`llm_gateway` 绝对不应该：

- 不直接跑模型；
- 不直接改配置文件；
- 不直接决定训练；
- 不直接决定模型晋升；
- 不直接写 finetune pool；
- 不直接更新 route map。

## 它在闭环中的正确位置

最终应该是：

```text
orchestration
   ↓
planning/context_builder
   ↓
llm_gateway.generate_structured_plan()
   ↓
planning/validator
   ↓
orchestration 执行被验证后的 plan
```

也就是说，大模型只能给建议：

```json
{
  "recommended_action": "call_expert_model",
  "preferred_expert_family": "under_segmentation_repair",
  "reason": "...",
  "confidence": 0.72
}
```

但真正是否执行，必须由：

```text
planning validator + rule guard + orchestration
```

共同决定。

## 建议新增/收口结构

```text
ITD_agent/llm_gateway/
  __init__.py
  client.py              # provider 统一调用
  schemas.py             # 结构化输出 schema
  prompt_builder.py      # prompt 构建
  response_parser.py     # JSON 解析与校验
  fallback.py            # LLM 失败时规则回退
  audit.py               # 调用审计
```

然后 `planning/scheduler` 中不应该自己处理 provider/base_url/model，而应该调用：

```python
llm_gateway.generate_plan(context, schema=ExpertRoutingPlan)
```

# 四、最终完整闭环应该长这样

不要再按 V1/V2/V3 暴露给用户，而是内部形成一个完整闭环：

```text
CLI
 ↓
orchestration
 ↓
input_layer
 ↓
data_processing
 ↓
segmentation/main_model
 ↓
evaluation_analysis
 ↓
planning + llm_gateway
 ↓
evolution/adaptive_inference
   ├─ ROI candidate
   ├─ expert task
   ├─ expert execution
   ├─ compare
   └─ fusion/rollback
 ↓
finetune_pool
   ├─ sample intake
   ├─ review
   ├─ quality filter
   └─ dataset export
 ↓
training_loop
   ├─ trigger
   ├─ pilot train
   ├─ replay guard
   ├─ formal train
   ├─ model registry
   └─ promotion candidate
 ↓
output_layer
 ↓
memory_store / knowledge_base / state
```

对应最终模块边界：

| 模块                   | 最终角色                     | 是否保留                   |
| ---------------------- | ---------------------------- | -------------------------- |
| `cli/`                 | 唯一用户入口                 | 新建/迁移                  |
| `orchestration/`       | 闭环总编排                   | 保留但大幅瘦身             |
| `input_layer/`         | 输入契约与主线 A/B gating    | 保留                       |
| `data_processing/`     | 数据预处理和在线场景状态     | 保留                       |
| `segmentation/`        | 主/专家模型统一执行器        | 保留                       |
| `evaluation_analysis/` | 评估、诊断、决策信号         | 保留                       |
| `planning/`            | 计划生成、专家路由、配置生成 | 保留                       |
| `llm_gateway/`         | LLM 统一调用与审计           | 必须补强                   |
| `evolution/`           | 推理期自适应闭环             | 保留但去掉 CLI/V1/V2 命名  |
| `finetune_pool/`       | 样本池、审查、导出           | 保留并吸收 V2 review       |
| `training_loop/`       | 受控训练和候选模型注册       | 保留                       |
| `output_layer/`        | 成果发布                     | 保留                       |
| `scripts/`             | 临时脚本/开发脚本            | 不允许被核心模块依赖       |
| `tools/`               | 通用工具                     | 可保留，但避免业务逻辑沉淀 |

# 七、最终一句话结论

你现在项目不是缺功能，而是**阶段性落地后的工程收口不够**。

最终修复目标不是继续加 V4，而是：

**把 V1/V2/V3 的阶段入口全部收敛成一个正式闭环；把 `cli`、`orchestration`、`evolution`、`finetune_pool`、`training_loop`、`llm_gateway` 的边界彻底固定；删除阶段性 shim 和重复脚本；让 LLM gateway 真正成为规划顾问层，而不是配置里有名字、流程里没存在感的摆设。**
